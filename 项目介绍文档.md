## 1. 一句话定位

> **FundPilot** 是一套面向 C 端用户的**基金研究与智能选基决策子系统**——在 Web 端把基金基础数据、净值走势、收益风险指标、基金经理画像、社区研究笔记和 AI 推荐解释整合在一起，辅助用户完成 **基金筛选 → 画像分析 → 横向对比 → 智能推荐 → 个人研究沉淀** 的全流程基金研究。

---

## 2. 项目背景

### 2.1 行业痛点

| # | 现状 | 痛点 |
| --- | --- | --- |
| 1 | 公开基金数量破万（C 端用户只能接触到 Top 100） | 选基盲选、信息过载 |
| 2 | 第三方 App 偏销售导向，深度数据隐藏在多级 Tab 后 | 研究门槛高 |
| 3 | 个人研究无法沉淀：零散的筛选条件、对比记录容易丢失 | 工作流断裂 |
| 4 | AI 投顾大多是黑盒，缺乏可解释性 | 用户不信任 |
| 5 | 国产数据库 KingbaseES 在金融信创领域落地，需要配套应用 | 课程级别 demo 稀缺 |

### 2.2 我们的解决方案

- **以"研究库"为中心**：把每位用户的研究动作（筛选条件、收藏、模拟组合、最近浏览）沉淀为可重入的工作流。
- **以"数据透明"为前提**：任何 AI 推荐都附数据来源 + 关键指标（收益、回撤、波动、标签），所见即所得。
- **以"国产化"为底座**：后端完整跑在 Spring Boot + MyBatis-Plus + KingbaseES 上，前端走 Vue 3 + Vite。
- **以"Dify"为枢纽**：AI 分析走 Dify 工作流，无 Key 自动 fallback 到本地说明，不影响主链路。

## 3. 核心亮点（四大产品能力）

### 3.1 基金筛选 + 基金画像

- 多维度条件筛选：基金类型 / 风险等级 / 收益区间 / 最大回撤上限 / 标签 / 关键词；
- 单基金详情页：`净值曲线` + `持仓饼图` + `风险雷达` + `阶段收益` + `归因` + `标签`;
- 智能评分（综合评分 + 维度分）和解释（标签+指标组合）。

### 3.2 基金对比

- 支持最多 N 只基金的横向比较（同视图叠加净值、最大回撤、波动率、综合评分）；
- 持有多只基金时可一键生成"对比记录"并命名归档；
- 同样可以触发 Dify AI 出对比解读。

### 3.3 我的研究池

每位登录用户的**个人研究空间**，含四个标签页：

| 子能力 | 含义 |
| --- | --- |
| 我的自选 | 一键收藏基金，长期跟踪 |
| 我的组合 | 多个基金组成模拟组合，沉淀配比思路 |
| 已保存筛选 | 把命中数 N 的筛选条件起个名字，下次一键重跑 |
| 最近浏览 | 倒序记录近 14 天看过的基金 |

> 这一块是整个产品的"粘性枢纽"——所有浏览/筛选/收藏动作都回流到这里。

### 3.4 基金社区 + AI 解读

- **社区**：用户发布研究笔记、点赞、评论、关注作者；支持精选 / 推荐作者 / 标签筛选；
- **AI 解读**：调用 Dify 工作流生成"为什么向该风险偏好用户推荐这只基金 / 这组对比"，无 API Key 时返回本地降级文案。

### 3.5 后台管理（管理员角色）

- 数据导入批次（`/admin`）+ 标签规则维护；
- 用户管理；
- 健康检查 + 数据库连通性探针（`/api/v1/health` `/api/v1/db-check`）。

## 8. 典型业务场景 Demo

### 场景 A：小白用户筛选稳健债基

1. 登录 → `/funds`；
2. 条件：基金类型 = "债券型"，最大回撤 ≤ 5%，近 1 年收益 ≥ 3%；
3. 把这条筛选命名为"稳健债基"保存 → `/my/research-pool` 中可一键重跑；
4. 进入某只看好的基金详情 → 看到 `风险雷达`、`持仓饼图`、`阶段指标` → **加入对比篮**；
5. 对比篮同时加入 3 只 → `/compare` 看到叠加净值 + AI 对比解读。

### 场景 B：研究员做新能源主题组合

1. `/funds`，条件：标签 = "新能源主题"，综合评分 80 分以上；
2. 选 5 只 → 创建组合"新能源策略 001"，记录初始权重；
3. 一周后再次访问 → 从"我的研究池 → 我的组合" 调出 → 调整权重；
4. 借助 Dashboard AI 解读，验证当前组合是否与"自身风险偏好（进取）"相符。

### 场景 C：内容运营管理社区精选

1. 用 admin 账号登录 → `/admin`；
2. 通过 `/api/v1/admin/import/batches` 查看历史导入批次，把优秀研究笔记置顶为精选；
3. 调整 `/api/v1/admin/tags/rules` 改变首页热门标签的排序。

## 9. 角色 & 权限模型

| 角色 | 路由示例 | 接口示例 | 备注 |
| --- | --- | --- | --- |
| **匿名访客** | `/`、`/login`、`/register` | 公开 | 仅能浏览营销页 |
| **普通用户** | `/dashboard`、`/funds`、`/my/research-pool`、`/community` | `/api/v1/my/research-pool/*` 仅自己 | 默认账号 `userId=1` |
| **管理员（角色码 ADMIN）** | 增加 `/admin` | `/api/v1/admin/**` | 通过 router.beforeEach 中的 `meta.adminOnly` 校验 |

> 角色码定义于 `fund_user.role_code`；权限校验通过 `SecurityUtils.currentUserOrNull()` / `assertSelfOrAdmin(...)` 完成。
> 当前阶段 Spring Security 全部 `permitAll`，仅是**占位**——后续可一行 `SecurityConfig` 配置补齐鉴权。

## 10. 安全与合规

- **密钥**：Dify API Key、JWT Secret、数据库密码、KingbaseES root 密码全部通过环境变量注入，**不落代码**；
- **依赖**：`pom.xml` 使用阿里云 Maven 镜像；`package-lock.json` 锁定版本；
- **API 设计**：所有列表分页、大整数 ID 序列化为字符串、Controller 不直连 Mapper、Entity 不直接对外；
- **Docker**：后端镜像（multi-stage `maven:3.9 → temurin:17-jre`）、前端镜像（`node:22-alpine → nginx:1.27-alpine`），最小化攻击面；
- **数据库**：使用 `psycopg2` 通过宿主端口连接 KingbaseES，容器内仅放无状态服务。

---

## 11. 性能与可扩展性策略

- **缓存层**：基金画像当前由 MyBatis-Plus 直查 KingbaseES，AI 解读已实现 `analysis_id` 级缓存命中（命中即不再调 Dify，节省 LLM 调用费）；
- **批量并发**：Dashboard 同时获取多个推荐解释时使用 `Promise.all + 2 路并发限流`；
- **前端**：Vite proxy 解决跨域；生产经 Nginx 反代 `/api/` 到后端；`gzip` 由 Nginx 默认开启；
- **可水平扩展**：后端无状态、可任意扩多实例；数据持久化全在 KingbaseES；
- **数据库演进**：所有业务逻辑落 SQL，没把规则写在 Java 里，便于未来迁移与 DBA 调优。
